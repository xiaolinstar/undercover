# WebSocket 实时通知系统 - 需求文档

## 1. 项目背景

项目正在从基于微信公众号的发牌器演进为标准的服务端项目，需要为微信小程序提供实时通知能力。采用 **WebSocket 推送通知 + HTTP API 拉取数据** 的混合架构模式。

**项目定位**：谁是卧底发牌器，专注于游戏状态管理和事件通知，不包含聊天、社交等复杂功能。

**部署策略**：
- **Phase 1（当前）**：单实例部署，架构简化
- **Phase 2（未来）**：多实例部署，水平扩展（待办）

## 2. 核心设计原则

### 2.1 职责分离
- **WebSocket**：仅用于轻量级事件通知（Event Notification）
- **HTTP API**：负责完整数据的查询和操作（Data Fetching & Mutation）

### 2.2 通信流程
```
1. 客户端通过 WS 连接到服务器并订阅房间
2. 房间内发生事件（如玩家加入、游戏开始、投票等）
3. 服务器通过 WS 推送轻量级事件通知
4. 客户端收到通知后，主动调用 HTTP API 获取最新数据
```

### 2.3 简化原则
- 不实现聊天室功能
- 不实现私聊功能
- 不实现复杂的消息队列
- 专注于游戏事件通知

## 3. 用户故事

### 3.1 作为小程序玩家
- **US-1.1**: 我希望在加入房间后能实时收到其他玩家加入的通知，这样我就知道何时可以开始游戏
- **US-1.2**: 我希望在游戏开始时立即收到通知，这样我可以查看我的词语
- **US-1.3**: 我希望在其他玩家投票后收到通知，这样我可以看到最新的投票进展
- **US-1.4**: 我希望在游戏结束时收到通知，这样我可以查看游戏结果
- **US-1.5**: 我希望在网络断开重连后能自动恢复房间订阅，不会错过消息

### 3.2 作为后端开发者
- **US-2.1**: 我希望 WebSocket 服务能够水平扩展，支持多实例部署
- **US-2.2**: 我希望能够方便地在业务代码中触发通知，而不需要关心底层实现
- **US-2.3**: 我希望能够监控 WebSocket 连接数、消息发送量等指标
- **US-2.4**: 我希望 WebSocket 断开不影响 HTTP API 的正常使用

### 3.3 作为小程序前端开发者
- **US-3.1**: 我希望 WebSocket 消息格式统一且易于解析
- **US-3.2**: 我希望能够通过 HTTP API 获取完整的房间状态，而不依赖 WS 消息
- **US-3.3**: 我希望在 WS 连接失败时能降级为轮询模式

## 4. 功能需求

### 4.1 WebSocket 连接管理

#### FR-1.1 建立连接
- 客户端通过 `/socket.io/` 端点建立 WebSocket 连接
- 连接建立时需要携带 JWT token 进行身份认证
- 服务器验证 token 后建立连接，返回连接成功消息

#### FR-1.2 房间订阅
- 客户端发送订阅消息：`{"action": "subscribe", "room_id": "8888"}`
- 服务器验证用户是否在该房间内
- 订阅成功后，用户将收到该房间的所有事件通知
- 单实例部署下，用户通常只在一个房间内

#### FR-1.3 取消订阅
- 客户端发送取消订阅消息：`{"action": "unsubscribe", "room_id": "8888"}`
- 服务器移除该用户对房间的订阅

#### FR-1.4 心跳保活
- 客户端每 30 秒发送一次 ping 消息
- 服务器响应 pong 消息
- 超过 60 秒未收到心跳，服务器主动断开连接

#### FR-1.5 断线重连
- 客户端检测到连接断开后自动重连（指数退避策略）
- 重连成功后自动重新订阅之前的房间

### 4.2 事件通知类型

所有 WebSocket 消息采用统一格式：
```json
{
  "event": "event_type",
  "room_id": "8888",
  "timestamp": 1708675200,
  "data": {
    // 轻量级提示数据
  }
}
```

#### FR-2.1 房间事件
- `room.player_joined`: 玩家加入房间
  ```json
  {
    "event": "room.player_joined",
    "room_id": "8888",
    "timestamp": 1708675200,
    "data": {
      "player_count": 4,
      "hint": "新玩家加入"
    }
  }
  ```

- `room.player_left`: 玩家离开房间
- `room.player_ready`: 玩家准备状态变更
- `room.disbanded`: 房间解散

#### FR-2.2 游戏事件
- `game.started`: 游戏开始（发牌完成）
  ```json
  {
    "event": "game.started",
    "room_id": "8888",
    "timestamp": 1708675200,
    "data": {
      "hint": "游戏已开始，请查看你的词语"
    }
  }
  ```

- `game.phase_changed`: 游戏阶段变更（描述 -> 投票 -> 结算）
- `game.player_eliminated`: 玩家被淘汰
- `game.ended`: 游戏结束
  ```json
  {
    "event": "game.ended",
    "room_id": "8888",
    "timestamp": 1708675200,
    "data": {
      "winner": "civilian",
      "hint": "游戏结束，平民获胜"
    }
  }
  ```

#### FR-2.3 投票事件
- `vote.submitted`: 有玩家提交了投票
  ```json
  {
    "event": "vote.submitted",
    "room_id": "8888",
    "timestamp": 1708675200,
    "data": {
      "voted_count": 3,
      "total_count": 5,
      "hint": "3/5 玩家已投票"
    }
  }
  ```

- `vote.completed`: 投票完成

### 4.3 HTTP API 增强

#### FR-3.1 房间状态同步接口（已存在，需增强）
- `GET /api/game/sync/{room_id}`
- 返回完整的房间状态，包括：
  - 房间基本信息
  - 所有玩家列表及状态
  - 当前游戏阶段
  - 投票进度（如果在投票阶段）
  - 当前用户的词语和角色
- 响应中包含 `version` 或 `updated_at` 字段用于版本控制

#### FR-3.2 批量状态查询（暂不实现）
- 未来扩展：支持一次查询多个房间的状态

### 4.4 服务端架构（单实例简化版）

#### FR-4.1 单实例部署
- 使用 flask-socketio 直接管理 WebSocket 连接
- 不使用 Redis Pub/Sub（单实例无需跨进程通信）
- 连接信息存储在内存中（进程内管理）

#### FR-4.2 通知服务抽象
- 创建 `NotificationService` 统一管理通知发送
- 业务代码调用简单接口：
  ```python
  notification_service.notify_room(
      room_id="8888",
      event="game.started",
      data={"hint": "游戏已开始"}
  )
  ```

#### FR-4.3 连接管理（简化版）
- 使用 flask-socketio 内置的 room 机制
- 不需要 Redis 存储连接映射
- 进程内存管理订阅关系

## 5. 非功能需求

### 5.1 性能要求
- **NFR-1.1**: 单个服务实例支持至少 500 个并发 WebSocket 连接（小程序场景）
- **NFR-1.2**: 事件通知延迟 < 500ms（从事件发生到客户端收到）
- **NFR-1.3**: HTTP API 响应时间 < 200ms (P95)

### 5.2 可靠性要求
- **NFR-2.1**: WebSocket 服务可用性 >= 99%
- **NFR-2.2**: 消息投递成功率 >= 95%（允许少量丢失，客户端可通过 HTTP 补偿）
- **NFR-2.3**: 支持优雅关闭，断开前通知客户端

### 5.3 可扩展性要求
- **NFR-3.1**: 架构设计支持未来扩展到多实例（但当前不实现）
- **NFR-3.2**: 支持动态添加新的事件类型，无需修改客户端代码

### 5.4 安全要求
- **NFR-4.1**: WebSocket 连接必须通过 JWT 认证
- **NFR-4.2**: 用户只能订阅自己加入的房间
- **NFR-4.3**: 防止消息重放攻击（timestamp 验证）

### 5.5 可观测性要求
- **NFR-5.1**: 记录所有 WebSocket 连接/断开事件
- **NFR-5.2**: 监控指标：当前连接数、消息发送量、错误率
- **NFR-5.3**: 结构化日志，便于问题排查

## 6. 技术选型

### 6.1 WebSocket 框架
- **选择**: `flask-socketio` + `python-socketio`
  - 与现有 Flask 架构无缝集成
  - 支持 WebSocket 和 Long Polling 降级
  - 单实例部署简单，无需额外配置
  - 成熟稳定，社区活跃

### 6.2 消息分发（单实例）
- **选择**: flask-socketio 内置的 room 机制
  - 进程内管理，无需 Redis Pub/Sub
  - 简单高效，延迟更低
  - 未来扩展：可配置 Redis 消息队列支持多实例

## 8. 验收标准

### AC-1: WebSocket 连接
- [ ] 客户端可以成功建立 WebSocket 连接
- [ ] 连接需要有效的 JWT token
- [ ] 无效 token 会被拒绝连接

### AC-2: 房间订阅
- [ ] 客户端可以订阅自己加入的房间
- [ ] 订阅不存在的房间会返回错误
- [ ] 订阅未加入的房间会返回权限错误
- [ ] 可以同时订阅多个房间

### AC-3: 事件通知
- [ ] 玩家加入房间时，房间内所有订阅者收到通知
- [ ] 游戏开始时，所有玩家收到通知
- [ ] 投票提交时，所有玩家收到投票进度通知
- [ ] 游戏结束时，所有玩家收到结果通知

### AC-4: HTTP API 同步
- [ ] 客户端收到 WS 通知后，调用 HTTP API 能获取最新数据
- [ ] HTTP API 返回的数据包含版本号或时间戳
- [ ] 即使未收到 WS 通知，HTTP API 也能返回正确数据

### AC-5: 多实例支持
- [ ] 部署 2 个 WebSocket 服务实例
- [ ] 客户端 A 连接实例 1，客户端 B 连接实例 2
- [ ] 在同一房间内，A 和 B 都能收到对方的操作通知

### AC-6: 断线重连
- [ ] 客户端断开连接后自动重连
- [ ] 重连后自动恢复房间订阅
- [ ] 重连期间的事件可以通过 HTTP API 补偿获取

### AC-7: 性能和稳定性
- [ ] 单实例支持 1000 个并发连接
- [ ] 事件通知延迟 < 500ms
- [ ] 连续运行 24 小时无内存泄漏

## 9. 风险和缓解措施

### 风险 1: WebSocket 连接不稳定
- **影响**: 客户端频繁断线重连，用户体验差
- **缓解**: 
  - 实现自动重连机制（指数退避）
  - 提供 HTTP 轮询降级方案
  - 优化心跳间隔

### 风险 2: 消息风暴
- **影响**: 大量客户端同时请求 HTTP API，服务器压力大
- **缓解**:
  - 客户端添加随机延迟（50-200ms）
  - Redis 缓存热点数据
  - 实现请求合并（批量查询）

### 风险 3: 跨实例消息延迟
- **影响**: Redis Pub/Sub 延迟导致通知不及时
- **缓解**:
  - 监控 Redis Pub/Sub 延迟
  - 考虑使用 Redis Streams（更可靠）
  - 客户端定期轮询兜底

### 风险 4: 内存泄漏
- **影响**: 长时间运行后服务器内存耗尽
- **缓解**:
  - 定期清理过期连接
  - 限制每个用户的订阅数量
  - 压力测试和内存监控

## 10. 后续优化方向

1. **消息优先级**: 重要消息（如游戏结束）优先投递
2. **离线消息**: 用户离线期间的消息存储，上线后推送
3. **消息去重**: 防止客户端收到重复通知
4. **智能降级**: 根据网络质量自动切换 WS/轮询
5. **消息压缩**: 对大量消息进行压缩传输

## 附录 B: 错误码定义

| 错误码 | 说明 | HTTP 状态码 |
|--------|------|-------------|
| `WS_AUTH_FAILED` | WebSocket 认证失败 | 401 |
| `WS_ROOM_NOT_FOUND` | 房间不存在 | 404 |
| `WS_PERMISSION_DENIED` | 无权限订阅该房间 | 403 |
| `WS_ALREADY_SUBSCRIBED` | 已经订阅该房间 | 400 |
| `WS_NOT_SUBSCRIBED` | 未订阅该房间 | 400 |
| `WS_INVALID_MESSAGE` | 消息格式错误 | 400 |
| `WS_RATE_LIMIT` | 请求频率过高 | 429 |
