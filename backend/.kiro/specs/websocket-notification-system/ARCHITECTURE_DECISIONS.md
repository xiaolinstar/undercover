# 架构决策记录

## 1. Redis 使用策略

### 决策：Redis 作为房间状态的主存储（非缓存）

**背景**：
- 原项目直接使用 Redis 存储房间状态
- 有担忧这是否是"不好的设计"

**分析**：
```
数据类型分类：
├── 持久化数据（应该用 MySQL）
│   ├── 用户资料（email, 注册时间等）
│   ├── 游戏历史记录
│   └── 订单、支付等核心业务数据
│
└── 临时数据（可以用 Redis 作为主存储）
    ├── 会话数据（Session）
    ├── 游戏房间状态（当前项目）✅
    ├── 实时排行榜
    └── 限流计数器
```

**决策**：
- ✅ **保持现状**：房间数据继续使用 Redis 作为主存储
- ✅ **游戏记录持久化**：游戏结束后写入 MySQL（已实现）
- ✅ **用户数据混合**：Redis 缓存会话，MySQL 持久化资料

**理由**：
1. 房间是临时数据（生命周期：几分钟到几小时）
2. 高频读写（每次操作都要更新状态）
3. 自动过期（TTL 2小时，无需手动清理）
4. 游戏结束后无价值（不需要持久化）
5. 性能优秀（Redis 比 MySQL 快 10-100 倍）

**不好的设计示例**（应该避免）：
```python
# ❌ 错误：用 Redis 存储用户核心数据
redis.set(f"user:{user_id}", json.dumps({
    "email": "user@example.com",
    "balance": 1000.00,  # 钱包余额！
    "created_at": "2024-01-01"
}))
```

**当前项目的设计**（合理）：
```python
# ✅ 正确：用 Redis 存储临时房间状态
redis.setex(f"room:{room_id}", 7200, json.dumps({
    "players": [...],
    "status": "playing",
    "current_round": 3
}))

# ✅ 正确：游戏结束后持久化到 MySQL
db.session.add(GameRecord(
    room_id=room_id,
    winner_team="civilian",
    ...
))
```

## 2. WebSocket 架构：单实例 vs 多实例

### 决策：先实现单实例，预留多实例扩展能力

**对比分析**：

| 维度 | 单实例（当前） | 多实例（未来） |
|------|---------------|---------------|
| **复杂度** | ⭐ 简单 | ⭐⭐⭐ 复杂 |
| **开发时间** | 1-2 周 | 3-4 周 |
| **运维成本** | 低 | 中 |
| **并发能力** | 500-1000 连接 | 5000+ 连接 |
| **单点故障** | 是 | 否 |
| **消息分发** | 进程内（快） | Redis Pub/Sub（慢 10ms） |
| **适用场景** | MVP、小规模 | 生产环境、大规模 |

**单实例架构**：
```
客户端 A ──WS──┐
客户端 B ──WS──┤──> Flask-SocketIO (单进程)
客户端 C ──WS──┘
```

**多实例架构**：
```
客户端 A ──WS──> 实例 1 ──┐
客户端 B ──WS──> 实例 2 ──┤──> Redis Pub/Sub
客户端 C ──WS──> 实例 3 ──┘
```

**决策**：
- ✅ **Phase 1**: 单实例部署（当前）
- ⏳ **Phase 2**: 多实例扩展（未来，待业务增长后）

**理由**：
1. **简化优先**：避免过度设计
2. **快速上线**：减少开发时间
3. **成本控制**：单实例运维成本低
4. **性能足够**：500 并发连接足够小程序初期使用
5. **易于扩展**：架构设计支持未来升级

**升级路径**：
```python
# 当前：单实例
socketio = SocketIO(app)

# 未来：多实例（只需添加一行配置）
socketio = SocketIO(
    app,
    message_queue='redis://localhost:6379/1'  # 启用 Redis Pub/Sub
)
```

## 3. Redis Pub/Sub 使用场景

### 决策：当前不使用，未来多实例时启用

**Redis Pub/Sub 适用场景**：
- ✅ 多实例 WebSocket 消息分发（你的未来场景）
- ✅ 实时聊天室
- ✅ 实时通知系统
- ✅ 微服务间事件通知
- ❌ 消息队列（不可靠，应该用 RabbitMQ/Kafka）
- ❌ 任务调度（应该用 Celery）

**为什么当前不需要**：
1. 单实例部署，无需跨进程通信
2. 进程内通信更快（< 1ms vs 10ms）
3. 减少依赖，降低复杂度
4. 节省 Redis 连接数

**什么时候需要**：
1. 并发连接数超过 1000
2. 需要高可用（避免单点故障）
3. 需要水平扩展

**实现对比**：

**单实例（当前）**：
```python
class NotificationService:
    def notify_room(self, room_id, event, data):
        # 直接发送，进程内通信
        self.socketio.emit(event, data, room=room_id)
```

**多实例（未来）**：
```python
class NotificationService:
    def notify_room(self, room_id, event, data):
        # 通过 Redis Pub/Sub 分发到所有实例
        # flask-socketio 自动处理，代码无需改变！
        self.socketio.emit(event, data, room=room_id)
```

**结论**：flask-socketio 的优势在于，升级到多实例时，业务代码无需修改！

## 4. 功能范围：专注发牌器，不做聊天室

### 决策：只实现游戏事件通知，不实现聊天功能

**功能对比**：

| 功能 | 发牌器（当前） | 聊天室（不做） |
|------|---------------|---------------|
| 玩家加入通知 | ✅ | ✅ |
| 游戏开始通知 | ✅ | ✅ |
| 投票通知 | ✅ | - |
| 文字聊天 | ❌ | ✅ |
| 表情包 | ❌ | ✅ |
| 私聊 | ❌ | ✅ |
| 消息历史 | ❌ | ✅ |
| 已读未读 | ❌ | ✅ |

**理由**：
1. **项目定位**：谁是卧底发牌器，不是社交应用
2. **复杂度控制**：聊天功能会增加 3-5 倍的开发量
3. **性能考虑**：聊天消息量远大于游戏事件
4. **存储成本**：聊天历史需要持久化存储
5. **审核风险**：聊天内容需要审核机制

**实现的事件**（轻量级）：
```python
# ✅ 实现：游戏事件通知
events = [
    "room.player_joined",    # 玩家加入
    "room.player_left",      # 玩家离开
    "game.started",          # 游戏开始
    "game.ended",            # 游戏结束
    "vote.submitted",        # 投票提交
]

# ❌ 不实现：聊天功能
not_implemented = [
    "chat.message",          # 聊天消息
    "chat.emoji",            # 表情包
    "chat.private_message",  # 私聊
]
```

## 5. 总结

### 核心架构决策

1. **Redis 使用**：
   - ✅ 房间状态作为主存储（合理）
   - ✅ 游戏记录持久化到 MySQL
   - ✅ 不改变现有设计

2. **WebSocket 架构**：
   - ✅ 单实例部署（当前）
   - ⏳ 多实例扩展（未来）
   - ✅ 不使用 Redis Pub/Sub（当前）

3. **功能范围**：
   - ✅ 专注游戏事件通知
   - ❌ 不实现聊天功能
   - ✅ 保持简单

### 技术栈

```
当前架构：
- Flask + flask-socketio（单实例）
- Redis（房间状态主存储）
- MySQL（游戏记录持久化）
- Eventlet（异步 IO）

未来扩展：
+ Redis Pub/Sub（多实例消息分发）
+ Nginx 负载均衡
+ Prometheus 监控
```

### 开发优先级

```
Phase 1（当前）：核心功能
├── WebSocket 连接和认证
├── 房间订阅机制
├── 游戏事件通知
└── HTTP API 增强

Phase 2（未来）：扩展功能
├── 多实例支持
├── 高级监控
└── 性能优化
```
